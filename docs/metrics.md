## 1. 基础计数

| 指标 | 字段 | 公式 | 含义 |
|------|------|------|------|
| 总请求数 | `total` | — | 压测发出的请求总数 |
| 成功数 | `success` | — | HTTP 200 且正常返回内容的请求数 |
| 失败数 | `fail` | — | 超时、限流、服务端错误等非正常返回 |
| 成功率 | `success_rate` | `success / total × 100` | 核心健康指标，正常服务应接近 100% |

### 判读

| 成功率 | 评估 | 排查方向 |
|--------|------|----------|
| 100% | ✅ 健康 | — |
| 99% ~ 100% | ⚠️ 偶发失败 | 查 `errors` 列表，通常为限流或临时抖动 |
| 95% ~ 99% | ⚠️ 偏高 | 服务端可能过载，降低并发重试 |
| < 95% | ❌ 异常 | 检查 API Key、网络、服务状态；大概率触发限流 |

## 2. 吞吐量

### QPS — Queries Per Second（每秒请求数）

```
QPS = success / wall_time
```

衡量 API 服务**每秒能处理多少个完整请求**。QPS 越高，并发处理能力越强。

### TPS — Tokens Per Second（每秒 Token 数）

```
TPS = total_output_tokens / wall_time
```

衡量 API **每秒能生成多少个 token**。TPS 越高，文本生成速度越快。

> **注意**：`wall_time` 是整个压测阶段的实际耗时（从第一个请求发出到最后一个请求返回），不是单个请求的延迟。

### QPS vs 并发数的关系

```
理论最大 QPS ≈ concurrency / latency_avg
```

| 现象 | 判断 |
|------|------|
| 实际 QPS 接近 `concurrency / latency_avg` | 服务端充分利用，并发打满 |
| 实际 QPS 远小于理论值 | 服务端有限流、排队严重 |
| 增大并发数后 QPS 不再增长 | **到达性能瓶颈**，继续加压只会增加延迟 |

### 判读（以通用 7B~70B 模型为参考）

| TPS | 评估 |
|-----|------|
| > 100 tok/s | 🟢 优秀，生成速度很快 |
| 50 ~ 100 tok/s | 🟡 良好 |
| 20 ~ 50 tok/s | 🟠 一般 |
| < 20 tok/s | 🔴 偏慢，可能是大模型或服务端资源不足 |

## 3. 延迟 Latency

延迟 = 单个请求从发起到**完整响应结束**的总耗时（秒）。

```
Latency = 响应结束时间 - 请求发起时间
```

### 百分位（Percentile）的含义

这是压测中最重要也最容易被误解的概念：

| 指标 | 通俗解释 |
|------|----------|
| **P50（中位数）** | 一半的请求比这个快，一半比这个慢。代表"普通用户体验" |
| **P95** | 95% 的请求快于这个值，只有 5% 更慢。代表"糟糕用户体验"的上限 |
| **P99** | 99% 的请求快于这个值。代表"极端情况" |
| **Avg（平均值）** | 所有请求的算术平均。**容易被长尾拉高，不能代表典型体验** |
| **Min** | 最快的那一次请求 |
| **Max** | 最慢的那一次请求 |

### 为什么 P50 和 P95 比 Avg 更重要？

```
假设 100 个请求：99 个耗时 1s，1 个耗时 100s

  Avg  = (99×1 + 100) / 100 = 1.99s   ← 被长尾严重拉高
  P50  = 1s                            ← 这才是大多数用户的真实体验
  P95  = 1s                            ← 依然正常
  P99  = 100s                          ← 精准捕获异常
```

> **结论：P50 代表典型体验，P95/P99 代表最差体验。关注这两个就够了，Avg 容易误导。**

### 长尾效应（Tail Latency）

```
P95 / P50 比值越大 → 长尾越严重 → 少数请求被严重拖延
```

| P95 / P50 | 评估 |
|-----------|------|
| < 1.5 | 🟢 延迟分布均匀，服务稳定 |
| 1.5 ~ 3 | 🟡 存在一定长尾，可接受 |
| 3 ~ 5 | 🟠 长尾明显，服务端可能存在排队或资源争抢 |
| > 5 | 🔴 长尾严重，建议排查或降低并发 |

### 延迟判读

延迟的好坏**取决于模型大小和业务场景**。以下为通用参考：

#### 流式场景（stream=true）

| 场景 | P50 参考 | P95 参考 |
|------|----------|----------|
| 实时对话 / 客服 | < 2s | < 5s |
| 内容生成 / 翻译 | < 10s | < 20s |
| 长文写作 / 代码生成 | < 30s | < 60s |

#### 非流式场景（stream=false）

| 场景 | P50 参考 | P95 参考 |
|------|----------|----------|
| 短问答 | < 3s | < 8s |
| 中等输出 | < 15s | < 30s |
| 长输出 | < 60s | < 120s |

## 4. 首 Token 延迟 TTFT

**TTFT = Time To First Token**，从请求发出到收到**第一个有内容的 token** 的时间。

> 仅流式模式有统计。非流式模式下不存在此指标。

### 为什么 TTFT 很重要？

TTFT 决定了用户**感知到的响应速度**。用户看到第一个字之前的时间，就是 TTFT。

```
一个完整的请求时间线：

  ├── TTFT ──┤├──── 生成时间 ────┤
  ↑           ↑                  ↑
  发请求    收到第一个字       全部输出完毕
                                  (Latency)
```

### TTFT 的组成

TTFT 主要包含：

1. **网络延迟** — 请求到达服务端的传输时间
2. **排队时间** — 请求在服务端等待 GPU 资源的时间（高并发时占比最大）
3. **Prefill 时间** — 模型处理输入 Prompt 的时间（Prompt 越长越慢）
4. **首 Token 生成** — 生成第一个 token 的推理时间

### 判读

| TTFT / Latency | 含义 |
|----------------|------|
| > 70% | 大部分时间花在等待首 Token，模型排队或 Prompt 处理是瓶颈 |
| 30% ~ 70% | 正常范围 |
| < 30% | 首 Token 很快，时间主要花在生成内容上（输出很长时常见） |

## 5. 每 Token 耗时 TPOT

**TPOT = Time Per Output Token**，衡量模型**每生成一个 token 平均需要多长时间**。单位通常为毫秒（ms）。

### 计算公式

```
流式:   TPOT = (latency - TTFT) / (output_tokens - 1)    ← 纯生成速度，剔除 prefill
非流式: TPOT = latency / output_tokens                   ← 含 prefill 的均值
```

> 流式模式下，TTFT 包含了 Prompt 处理（Prefill）时间，将其扣除后再算 TPOT 才能反映模型的**纯粹生成速度**。非流式模式无法区分 prefill 和生成阶段，因此 TPOT 为整体均值。

### TPOT vs TPS 的区别

这是最容易混淆的两个指标：

| 指标 | 维度 | 公式 | 回答的问题 |
|------|------|------|------------|
| **TPS** | 系统级 | `total_output_tokens / wall_time` | 服务端整体吞吐能力如何？ |
| **TPOT** | 请求级 | `(latency - TTFT) / (output_tokens - 1)` | 单个用户体感每 token 等多久？ |

```
示例：并发 50，wall_time=10s，总输出 5000 token，单个请求输出 100 token，latency=2s, TTFT=0.3s

  TPS   = 5000 / 10  = 500 tok/s     ← 看起来很快
  TPOT  = (2 - 0.3) / (100 - 1) = 17.2 ms/tok ≈ 58 tok/s  ← 用户真实体感
```

> **TPS 会随并发数线性增长，TPOT 反映的是单用户体验。两者缺一不可。**

### 判读

| TPOT (ms/tok) | 等效生成速度 | 评估 |
|---------------|-------------|------|
| < 10 ms | > 100 tok/s | 🟢 优秀，流式输出几乎无感知延迟 |
| 10 ~ 20 ms | 50 ~ 100 tok/s | 🟡 良好，阅读体验流畅 |
| 20 ~ 50 ms | 20 ~ 50 tok/s | 🟠 一般，能感知到逐字输出 |
| 50 ~ 100 ms | 10 ~ 20 tok/s | 🔴 偏慢，明显卡顿感 |
| > 100 ms | < 10 tok/s | ⚠️ 很慢，用户等待焦虑 |

### 实战场景

| 场景 | 看什么 | 为什么 |
|------|--------|--------|
| 对比不同服务商的同款模型 | TPOT P50 | 模型相同，TPOT 差异反映基础设施差距 |
| 发现并发过载 | TPOT P95 随并发增大是否暴涨 | 排队时间最终会反映在 token 间隔上 |
| 评估流式用户体验 | TPOT Avg | 比 TPS 更准确地反映"打字机效果"的流畅度 |
| 长输出场景（代码生成、写作） | TPOT P95 | 输出 token 多，单 token 延迟波动对总耗时影响大 |

## 6. Token 统计

| 指标 | 含义 |
|------|------|
| `total_input_tokens` | 所有请求消耗的输入 token 总数（Prompt） |
| `total_output_tokens` | 所有请求生成的输出 token 总数（Completion） |

### 用途

- **成本估算**：`输入价格 × total_input_tokens + 输出价格 × total_output_tokens`
- **平均输出长度**：`total_output_tokens / success`，评估模型是否稳定输出
- **输入/输出比**：判断压测 Prompt 的 token 消耗结构

> 注意：非流式模式下 input_tokens 来自 API 返回的 `usage` 字段；流式模式下为粗略估算（字符数÷2），不够精确。

## 7. 错误信息

`errors` 数组记录失败的请求详情，每条包含：

| 字段 | 含义 |
|------|------|
| `code` | HTTP 状态码（0 表示请求未到达服务端，如超时/连接失败） |
| `msg` | 错误消息（截断到 200 字符） |

### 常见错误码

| 状态码 | 典型原因 | 处理建议 |
|--------|----------|----------|
| 0 | 连接超时、DNS 失败、网络不通 | 检查网络和 base_url |
| 429 | 限流（Rate Limit） | 降低并发数或增加请求间隔 |
| 500 | 服务端内部错误 | 重试，持续出现则联系服务方 |
| 502/503 | 服务不可用/网关错误 | 等待后重试 |
| 401 | API Key 无效 | 检查 api_key |
| 404 | 接口路径不存在 | 检查 base_url 是否正确 |

```
                        ┌──────────────────────────┐
                        │      压测配置              │
                        │  concurrency × requests   │
                        └──────────┬───────────────┘
                                   │
                    ┌──────────────┴──────────────┐
                    │                             │
                    ▼                             ▼
            ┌──────────────┐              ┌──────────────┐
            │   吞吐量      │              │   延迟        │
            │  QPS / TPS   │◄─────────────│  Latency     │
            └──────┬───────┘   反比关系    └──────┬───────┘
                   │                             │
                   │                    ┌────────┴────────┐
                   │                    │                 │
                   │                    ▼                 ▼
                   │            ┌──────────────┐  ┌──────────────┐
                   │            │   总延迟      │  │  首 Token    │
                   │            │  P50/P95     │  │  TTFT        │
                   │            └──────────────┘  └──────────────┘
                   │
                   ▼
            ┌──────────────┐
            │   成功率      │
            │ success_rate │
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │   错误列表    │
            │   errors     │
            └──────────────┘
```

### 核心关系

```
QPS ↑  ←→  Latency ↓        （反比：延迟越低，吞吐越高）
并发 ↑  →  QPS ↑, Latency ↑ （正比到瓶颈后反转）
并发 ↑  →  TTFT ↑           （排队时间增加）
并发 ↑↑ →  success_rate ↓   （过载导致失败）
```

## 9. 评估标准速查

### 一次好的压测应该关注什么？

按优先级排序：

| 优先级 | 指标 | 看什么 |
|--------|------|--------|
| 🔴 第一 | `success_rate` | 必须 ≥ 99%，否则结果无效 |
| 🟠 第二 | `latency_p95` | 是否满足业务 SLA |
| 🟡 第三 | `qps` / `tps` | 吞吐是否满足业务需求 |
| 🟢 第四 | `ttft_avg` / `tpot_p50` | 首字延迟 + 生词速度，直接决定用户体验 |
| 🔵 参考 | `latency_p50` / `tpot_p95` | 典型体验 + 极端卡顿情况 |

### 压测报告模板

```
模型: xxx   并发: xx   请求数: xxx
─────────────────────────────────
✅ 成功率:   100%           ← 健康
🚀 QPS:      x.xx req/s    ← 每秒处理能力
🚀 TPS:      xx.x tok/s    ← 文本生成速度
⏱️ 延迟 P50: x.xx s        ← 典型体验
⏱️ 延迟 P95: x.xx s        ← 最差 5% 体验
🔤 TTFT P50: x.xx s        ← 等多久看到第一个字
⚡ TPOT P50: xx.x ms       ← 每个 token 生成间隔
📊 长尾比:   x.xx          ← P95/P50，<2 为健康
```

---

> **持续更新**：不同模型、不同服务商的指标基准值差异很大。建议每次压测后记录结果，积累自己的性能基线数据。
