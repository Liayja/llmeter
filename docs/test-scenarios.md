## 0. 压测前必读：避免无效结果的 6 个原则

压测最大的陷阱是**测出来的数字不可信**。先记住这 6 条：

| # | 原则 | 原因 |
|---|------|------|
| 1 | **固定变量，只改一个** | 同时改并发和 max_tokens，无法判断是谁影响了延迟 |
| 2 | **预热后再测** | 首次请求有冷启动（建连、DNS、服务端加载），本工具已内置预热 |
| 3 | **请求数足够大** | 样本太小，P95/P99 没有统计意义（见原则 5） |
| 4 | **固定输出长度** | 用 `--max-tokens` 限制输出，否则延迟受内容长度随机干扰 |
| 5 | **请求数 ≥ 并发数 × 10** | 保证每个并发槽位至少跑 10 轮，百分位才稳定 |
| 6 | **同条件多跑几次取中位** | 网络抖动会让单次结果失真，重复 3 次看一致性 |

> ⚠️ **关于 P99**：要让 P99 有意义，至少需要 100 个请求（100 个里第 99 个）。只跑 20 个请求时，P95 和 P99 会是同一个值（你之前的测试就是如此），此时 P99 不可信。

## 1. 压测方法论：四步流程

```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│ ① 基准测试   │──▶│ ② 阶梯加压   │──▶│ ③ 容量确认   │──▶│ ④ 稳定性验证 │
│ 单并发摸底   │   │ 逐步找拐点   │   │ 锁定最优并发 │   │ 长时间无衰减 │
└─────────────┘   └─────────────┘   └─────────────┘   └─────────────┘
   建立基线          找性能瓶颈        确定生产配置        验证可靠性
```

每一步的目的：

1. **基准测试** — 在无压力下测出单请求的真实延迟，作为后续对比的"地板值"
2. **阶梯加压** — 逐步增加并发，观察 QPS 何时不再增长、延迟何时陡增 → 找到**拐点**
3. **容量确认** — 在拐点附近的最优并发上充分测试，确定生产环境配置
4. **稳定性验证** — 长时间运行，确认没有内存泄漏、限流累积、性能衰减

## 2. 核心测试场景

### 2.1 基准测试（Baseline）

**目的**：测出单请求的纯净延迟，建立对比基线。

| 参数 | 取值 | 说明 |
|------|------|------|
| 并发 `-c` | **1** | 单并发，无排队干扰 |
| 请求数 `-n` | 20 ~ 30 | 足够算出 P50 |
| max-tokens | 固定（如 256） | 消除输出长度变量 |

```bash
python bench.py chat -f config/config.yaml -c 1 -n 30 --max-tokens 256
```

**关注**：`latency_p50`、`ttft_p50`。这是该模型在**无压力下的最佳表现**，后续所有加压结果都和它对比。

---

### 2.2 阶梯加压（Ramp-up / 找拐点）

**目的**：最重要的场景。逐步加压，找到性能拐点。

固定 `max-tokens`，逐步翻倍并发：

```bash
# 保持每个并发槽跑 ~20 轮，请求数随并发同步增大
python bench.py chat -f config/config.yaml -c 1   -n 30   --max-tokens 256
python bench.py chat -f config/config.yaml -c 5   -n 100  --max-tokens 256
python bench.py chat -f config/config.yaml -c 10  -n 200  --max-tokens 256
python bench.py chat -f config/config.yaml -c 20  -n 400  --max-tokens 256
python bench.py chat -f config/config.yaml -c 50  -n 1000 --max-tokens 256
python bench.py chat -f config/config.yaml -c 100 -n 2000 --max-tokens 256
```

把结果填进对比表：

| 并发 | QPS | TPS | P50 延迟 | P95 延迟 | TTFT P50 | 成功率 |
|------|-----|-----|----------|----------|----------|--------|
| 1    |     |     |          |          |          |        |
| 5    |     |     |          |          |          |        |
| 10   |     |     |          |          |          |        |
| 20   |     |     |          |          |          |        |
| 50   |     |     |          |          |          |        |
| 100  |     |     |          |          |          |        |

**拐点判断**（详见 [第 4 节](#4-结果判读与拐点识别)）：QPS 不再增长 / 延迟陡增 / 成功率下降，三者之一出现即为拐点。

---

### 2.3 容量测试（Capacity）

**目的**：在拐点附近确定生产环境的最优并发数。

假设阶梯加压发现拐点在并发 50 附近，就在它周围精细测试：

```bash
python bench.py chat -f config/config.yaml -c 40 -n 800 --max-tokens 256
python bench.py chat -f config/config.yaml -c 45 -n 900 --max-tokens 256
python bench.py chat -f config/config.yaml -c 50 -n 1000 --max-tokens 256
python bench.py chat -f config/config.yaml -c 55 -n 1100 --max-tokens 256
```

**最优并发 = 满足 SLA（如 P95 < 5s）前提下，QPS 最高的那个并发数。**

---

### 2.4 稳定性测试（Soak / 长时间）

**目的**：验证长时间运行下性能不衰减、无累积性限流。

用容量测试确定的最优并发，跑大量请求：

```bash
python bench.py chat -f config/config.yaml -c 50 -n 5000 --max-tokens 256
```

**关注**：
- 成功率是否始终保持 100%（限流通常在持续压力下才触发）
- 对比前后段延迟是否稳定（可看 `details` 数组前后部分）

---

### 2.5 不同 Prompt 长度对比

**目的**：评估输入长度对 TTFT（Prefill 阶段）的影响。

准备三个不同长度的 prompt 文件：

```bash
# 短输入（~50 token）
python bench.py chat -f config/config.yaml --prompt-file prompts/short.txt -c 10 -n 200 --max-tokens 256

# 中等输入（~1k token）
python bench.py chat -f config/config.yaml --prompt-file prompts/medium.txt -c 10 -n 200 --max-tokens 256

# 长输入（~4k token，模拟 RAG / 长文档）
python bench.py chat -f config/config.yaml --prompt-file prompts/long.txt -c 10 -n 200 --max-tokens 256
```

**关注**：输入越长，`ttft` 应该越大（Prefill 计算量增加）。如果 TTFT 随输入长度线性飙升，说明 Prefill 是瓶颈。

---

### 2.6 流式 vs 非流式对比

**目的**：评估两种模式的体验差异。

```bash
# 非流式（拿到完整结果才返回）
python bench.py chat -f config/config.yaml -c 10 -n 200 --max-tokens 512

# 流式（逐 token 返回，可测 TTFT）
python bench.py chat -f config/config.yaml -c 10 -n 200 --max-tokens 512 --stream
```

**关注**：
- 非流式：只有总 `latency`，无 TTFT
- 流式：可看 `ttft`，用户体验更好（首字快），但总延迟可能略高
- 非流式的 token 统计更准确（来自 API 的 `usage` 字段）

---

### 2.7 极限/破坏性测试（Stress）

**目的**：找到服务的崩溃边界，确认错误处理是否优雅。

故意超过拐点，加压到失败：

```bash
python bench.py chat -f config/config.yaml -c 200 -n 2000 --max-tokens 256
python bench.py chat -f config/config.yaml -c 500 -n 5000 --max-tokens 256
```

**关注 `errors` 列表**：
- 出现 429 → 限流阈值，记录此时的并发数
- 出现 500/503 → 服务端过载
- 大量 code=0 → 超时，可能需要调大 `-t`

> ⚠️ 破坏性测试可能触发服务商的风控/封禁，**仅在自己可控的服务或获得授权的环境下进行**。

## 3. 参数设计指南

### 各参数对结果的影响

| 参数 | 主要影响 | 设计建议 |
|------|----------|----------|
| `-c` 并发 | QPS、延迟、成功率 | 核心变量，阶梯式递增 1→5→10→20→50→100 |
| `-n` 请求数 | 结果统计可信度 | **≥ 并发 × 10**；要看 P99 则 ≥ 100 |
| `--max-tokens` | 输出长度→总延迟 | **务必固定**，对比测试时保持一致 |
| `--temperature` | 输出随机性 | 压测建议固定（如 0.7），保证可复现 |
| `--prompt-file` | 输入长度→TTFT | 单独测试输入长度影响时变化 |
| `--stream` | 是否测 TTFT | 测首字延迟必开；测纯吞吐可关 |
| `-t` 超时 | 失败判定 | 默认 30s，长输出场景需调大 |

### 请求数与并发数的配比参考

| 并发 `-c` | 推荐请求数 `-n` | 说明 |
|-----------|----------------|------|
| 1 | 30 | 基准 |
| 5 | 100 | |
| 10 | 200 | |
| 20 | 400 | |
| 50 | 1000 | |
| 100 | 2000 | 每槽 20 轮 |

## 4. 结果判读与拐点识别

随着并发增加，系统会经历三个阶段：

```
 QPS
  ▲
  │              ┌──────────── 饱和区：QPS 不再增长，延迟陡增
  │           ╱──┘           （继续加压只会让延迟更糟）
  │        ╱
  │     ╱   ◄──── 拐点（最优并发）
  │   ╱
  │ ╱   ◄──── 线性区：QPS 随并发线性增长，延迟平稳
  │╱
  └────────────────────────────────▶ 并发数
```

| 阶段 | 特征 | 应对 |
|------|------|------|
| **线性区** | 并发↑ → QPS↑，延迟基本不变，成功率 100% | 服务有余量，可继续加压 |
| **拐点** | QPS 增长放缓，延迟开始明显上升 | **生产配置选这里** |
| **饱和区** | QPS 停滞甚至下降，P95 陡增，开始出现失败 | 已过载，需降并发 |

### 拐点的三个信号（任一出现即到拐点）

1. **QPS 信号**：并发翻倍，QPS 增长 < 20%
2. **延迟信号**：P95 相比上一档增长 > 50%
3. **错误信号**：`success_rate` 开始 < 100%（出现 429）

## 5. 常见误区

| ❌ 误区 | ✅ 正确做法 |
|---------|-----------|
| 只跑一次就下结论 | 同条件跑 3 次，看一致性 |
| 请求数太少（如 20）就看 P99 | 看 P99 至少 100 个请求 |
| 输出长度不固定 | 用 `--max-tokens` 锁定 |
| 只看平均延迟 | 重点看 P50（典型）和 P95（最差） |
| 一上来就打 500 并发 | 从低并发阶梯加压，找拐点 |
| 本地网络差还测延迟 | 在靠近服务的稳定网络环境测，或区分网络延迟 |
| 用不同 Prompt 对比并发 | 对比并发时 Prompt 必须完全一致 |
| 忽略成功率直接看 QPS | 成功率 < 99% 时，QPS 数据无意义 |

## 6. 完整压测 Checklist

压测前：

- [ ] 确认 `base_url` / `api_key` / `model` 正确
- [ ] 固定 `--max-tokens` 和 `--temperature`
- [ ] 准备好对照用的 Prompt（长度一致）
- [ ] 确认网络环境稳定

执行中：

- [ ] 先跑基准（`-c 1`）建立基线
- [ ] 阶梯加压，每档请求数 ≥ 并发 × 10
- [ ] 每档记录到对比表
- [ ] 关注成功率，< 99% 立即停止加压

压测后：

- [ ] 识别拐点（QPS / 延迟 / 错误三信号）
- [ ] 确定最优并发数
- [ ] 同条件复测 3 次确认一致性
- [ ] 归档结果 JSON，记录测试条件

---

> **一句话总结**：固定所有变量、只动并发、从小到大阶梯加压、请求数给够、关注 P95 和成功率、找到 QPS 不再涨的拐点——这个拐点就是你要的答案。
