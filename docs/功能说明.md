## 1. 架构概览

```
bench.py          ← CLI 入口，argparse 子命令路由
src/
├── engine.py      ← 异步压测引擎，负责 HTTP 请求、指标聚合
└── output.py      ← 终端打印 + 文件保存（JSON / Markdown / CSV）
config/            ← 配置文件目录
├── config.yaml    ← 单次压测配置文件
└── scenarios.yaml ← 多场景对比配置文件
prompts/           ← 测试 Prompt 模板（短/中/长输入 + 业务场景）
docs/
├── metrics.md     ← 指标手册
├── test-scenarios.md ← 测试场景方法论
└── 功能说明.md     ← 本文档
bench_results/     ← 结果输出目录，按日期分文件夹
```

数据流：

```
配置文件 / CLI 参数
      │
      ▼
  bench.py ──► build_engine_config()
      │
      ▼
  LLMBench.run()
      ├── 预热 (2 次请求)
      ├── 并发压测 (asyncio + httpx)
      └── aggregate() ──► 20+ 聚合指标
      │
      ▼
  output.py
      ├── print_report() ──► 终端 rich 表格
      ├── save_results() ──► JSON 文件
      ├── print_comparison() ──► 终端对比表
      └── save_comparison() ──► Markdown + CSV
```

## 2. CLI 双模式

### 启动方式

```bash
# 单次压测
python bench.py chat -f config/config.yaml

# 多场景对比压测
python bench.py scenarios -f config/scenarios.yaml
```

使用 `argparse` 子命令实现，两个模式互不冲突。参数优先级：

```
命令行 > 配置文件 > 默认值
```

### chat — 单次压测

```bash
# 纯配置文件
python bench.py chat -f config/config.yaml

# 纯命令行
python bench.py chat -b https://api.openai.com -k sk-xxx -m gpt-4o -c 10 -n 100

# 配置文件 + 命令行覆盖（并发数用 50，其余从配置文件取）
python bench.py chat -f config/config.yaml -c 50 --stream
```

可用参数：

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `-f, --config` | 配置文件路径 (.yaml / .json) | 无 |
| `-b, --base-url` | API 基础 URL | 无 |
| `-k, --api-key` | API Key | 无 |
| `-m, --model` | 模型名称 | 无 |
| `-p, --prompt` | 压测 Prompt 文本 | `Hello, world!` |
| `--prompt-file` | 从文件加载 Prompt | 无 |
| `-c, --concurrency` | 并发数 | 10 |
| `-n, --requests` | 总请求数 | 100 |
| `-t, --timeout` | 单请求超时（秒） | 30 |
| `-o, --output-dir` | 结果输出目录 | `bench_results` |
| `--max-tokens` | 最大生成 token 数 | 2048 |
| `--temperature` | 生成温度 | 0.7 |
| `--stream` | 启用流式响应 | 关闭 |

### scenarios — 多场景对比压测

```bash
python bench.py scenarios -f config/scenarios.yaml
```

- 按顺序执行配置文件中的多个场景
- 每场景可独立覆盖并发数、请求数、Prompt、`max_tokens` 等参数
- 场景合并优先级：**场景值 > defaults > 顶层公共值 > 全局默认**
- 全部执行完成后自动生成对比表（终端 + Markdown + CSV）
- 任一场景有失败则返回非 0 退出码

## 3. 压测引擎

文件：[src/engine.py](src/engine.py)

### 请求模型

- 纯 Chat Completions，`POST /v1/chat/completions`
- 支持**流式**（`stream=true`，SSE 解析）和**非流式**（`stream=false`）
- HTTP/2 + 异步并发，基于 `httpx.AsyncClient` + `asyncio.Semaphore`

### 预热机制

2 次预热请求在主压测之前发出，复用同一客户端连接池，避免冷启动（DNS 解析、TLS 握手、服务端模型加载）影响压测结果。

### 重试逻辑

- 瞬时错误自动指数退避重试：`429`（限流）、`500`、`502`、`503`、连接失败（状态码 0）
- 默认最多重试 2 次，退避因子 1.0s，间隔公式：`backoff × 2^attempt`
- 持久性错误（如 401、404）不重试，立即返回失败
- 可通过 `retries` 和 `retry_backoff` 配置

### 延迟计时

`t0` 在获取信号量**之后**开始计时——延迟只反映 API 响应时间（网络 + 服务端处理），不含客户端排队等待时间。这样 P50/P95/P99 准确代表服务端性能。

### Token 计数

| 模式 | 优先级 | 方法 |
|------|--------|------|
| 流式 | 第一 | `stream_options={"include_usage": true}` 获取 API 返回的准确 usage |
| 流式 | 第二 | `tiktoken` 本地编码（`cl100k_base`） |
| 流式 | 第三 | 中英文字符估算：CJK 字符 ≈ 1.5 char/token，其他 ≈ 4 char/token |
| 非流式 | — | 直接从响应的 `usage.prompt_tokens` / `usage.completion_tokens` 获取 |

> 相比旧版 `text_len // 2`，token 计数的准确度大幅提升。安装 `tiktoken` 可获得最高精度。

### 流式解析

支持多种 chunk 格式，适配不同服务商：

- `choices[0].delta.content`（OpenAI / DeepSeek 标准格式）
- `choices[0].delta.text`（部分模型）
- `choices[0].message.content`（某些非标准实现）
- 处理 usage chunk（`stream_options` 返回的最终统计）

### 指标聚合（共 20+ 个指标）

| 类别 | 指标 | 说明 |
|------|------|------|
| **基础** | `total` | 压测发出的请求总数 |
| | `success` | HTTP 200 且正常返回内容的请求数 |
| | `fail` | 超时、限流、服务端错误等非正常返回 |
| | `success_rate` | `success / total × 100` |
| **吞吐** | `qps` | 每秒请求数 (Queries Per Second) |
| | `tps` | 每秒 Token 数 (Tokens Per Second) |
| **Token** | `total_input_tokens` | 所有请求消耗的输入 token 总数 |
| | `total_output_tokens` | 所有请求生成的输出 token 总数 |
| **延迟** | `latency_avg` | 平均延迟 |
| | `latency_p50` | 中位数延迟（典型用户体验） |
| | `latency_p95` | 95 分位延迟（最差 5% 体验） |
| | `latency_p99` | 99 分位延迟（极端情况） |
| | `latency_min` | 最快请求 |
| | `latency_max` | 最慢请求 |
| **TTFT** | `ttft_avg` | 平均首 Token 延迟 |
| | `ttft_p50` | 首 Token 延迟中位数 |
| | `ttft_p95` | 首 Token 延迟 95 分位 |
| **TPOT** | `tpot_avg` | 平均每 Token 生成耗时 (ms) |
| | `tpot_p50` | 每 Token 耗时中位数 |
| | `tpot_p95` | 每 Token 耗时 95 分位 |
| **错误** | `errors` | 失败请求摘要（前 10 条，含状态码和错误消息） |

> 百分位数使用**线性插值**计算：`s[f] + c × (s[f+1] - s[f])`，小样本下也比简单索引法更精确。

## 4. 输出模块

文件：[src/output.py](src/output.py)

### 终端打印

| 函数 | 用途 |
|------|------|
| `print_report(metrics, model, engine_type)` | 单次压测结果 |
| `print_comparison(rows, model)` | 多场景对比表 |

- 优先使用 `rich` 库渲染彩色表格（自动检测是否安装）
- `rich` 不可用时回退到纯文本格式，列宽自动适配内容
- 成功率 < 99% 整行标红告警

### 文件保存

| 模式 | 函数 | 格式 | 内容 |
|------|------|------|------|
| 单次压测 | `save_results()` | JSON | `meta`（配置参数）+ `summary`（聚合指标）+ `details`（每条请求明细） |
| 多场景对比 | `save_comparison()` | Markdown | 对比表格，可直接粘贴到文档 |
| 多场景对比 | `save_comparison()` | CSV | 原始数值，`utf-8-sig` 编码，Excel 直接打开 |

输出按日期分文件夹：`{output_dir}/YYYY-MM-DD/`

## 5. 配置系统

### 支持格式

- **YAML**（需 `pyyaml`）— 推荐
- **JSON** — 备选

### 单次压测配置 (`config.yaml`)

直接定义全部参数：

```yaml
base_url: https://api.deepseek.com
api_key: sk-xxx
model: deepseek-v4-pro
concurrency: 10
requests: 100
timeout: 30
max_tokens: 2048
temperature: 0.7
stream: true
prompt: "以春为题写一首五言绝句"
output_dir: bench_results
```

> **`base_url` 格式说明：** 只需填写 API 的根地址即可，不要加路径或尾部斜杠。程序会自动拼接 `/v1/chat/completions`（OpenAI 标准接口格式）。
>
> | ✅ 正确 | ❌ 错误 |
> |---|---|
> | `https://api.openai.com` | `https://api.openai.com/v1/chat/completions` |
> | `https://api.deepseek.com` | `https://api.deepseek.com/v1/` |
> | `https://your-proxy.com` | `https://your-proxy.com/` |

### 多场景配置 (`scenarios.yaml`)

三层结构：顶层公共参数 + `defaults` 默认值 + `scenarios` 列表：

```yaml
# 顶层公共参数（所有场景共享）
base_url: https://api.deepseek.com
api_key: sk-xxx
model: deepseek-v4-pro
output_dir: bench_results

# 默认参数（场景未指定时使用）
defaults:
  concurrency: 10
  requests: 100
  timeout: 30
  stream: true
  max_tokens: 256

# 场景列表（每个场景可覆盖任意参数）
scenarios:
  - name: 短输入
    prompt_file: prompts/short.txt

  - name: 智能客服
    prompt_file: prompts/biz-customer-service.txt
    concurrency: 20
    requests: 200
    max_tokens: 200

  - name: 代码生成
    prompt_file: prompts/biz-code-generation.txt
    max_tokens: 1024
    timeout: 60
```

合并优先级：**场景值 > defaults > 顶层公共值 > 全局默认**

## 6. 测试 Prompt 集

预置 6 个测试 Prompt，覆盖输入长度分级和真实业务场景：

| 文件 | 场景 | 输入长度 | 输出特征 | 建议 max_tokens |
|------|------|----------|----------|-----------------|
| `prompts/short.txt` | 短输入 | ~30 token | 固定 | 256 |
| `prompts/medium.txt` | 中输入 | ~300 token | 固定 | 256 |
| `prompts/long.txt` | 长输入 (RAG 模拟) | ~2000 token | 固定 | 256 |
| `prompts/biz-customer-service.txt` | 智能客服 | 短 | 短 | 200 |
| `prompts/biz-translation.txt` | 翻译 | 中 | 中 | 300 |
| `prompts/biz-code-generation.txt` | 代码生成 | 短 | 长 | 1024 |

> 前三组用于对比输入长度对 TTFT / Prefill 性能的影响（输出固定，唯一变量是输入长度）。后三组用于评估真实业务负载下的综合表现。
